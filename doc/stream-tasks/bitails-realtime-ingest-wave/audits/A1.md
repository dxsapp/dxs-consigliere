# A1 Audit

## Decision

The wave is sufficient to treat Bitails as the practical default live realtime ingest source for managed-scope ingest.

## Why

- runtime source selection for `realtime_ingest` now flows through `SourceCapabilityRouting`
- Bitails live ingest uses the existing `ITxMessageBus` and downstream `TransactionFilter` path
- startup bootstrap respects policy-driven ownership for node/ZMQ instead of starting mempool and block bootstrap unconditionally
- managed-scope subscription planning remains centralized in the Bitails adapter seam and is not reimplemented inside runtime orchestration

## Residual Role Of JungleBus

JungleBus remains available for:
- optional advanced/manual realtime ingest when selected by routing
- existing missing-data or historical-assist flows outside this bounded wave

This wave does not demote JungleBus out of the repository. It removes the hard-coded assumption that JungleBus is the only live realtime source.

## Transport Limitation

Bitails realtime planning now understands both websocket and ZMQ transport intent, but the live runtime consumer in this wave is websocket-only.

That limitation is explicit:
- ZMQ planning is supported for bootstrap and future runtime expansion
- `BitailsRealtimeIngestRunner` currently throws for non-websocket runtime plans instead of silently pretending support
