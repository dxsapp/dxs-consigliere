# A1 Audit

## Verdict

`first-run-setup-wizard-wave` is acceptable to close.

The product no longer depends on the provider-first `/providers` page for first-run onboarding.
A new operator now gets a capability-first bootstrap flow that configures:
- admin access
- raw transaction source
- REST fallback
- realtime ingest

## What Was Proved

- setup/bootstrap state is persisted in DB
- admin bootstrap credentials are no longer sourced from static appsettings
- `/api/setup/status`, `/api/setup/options`, and `/api/setup/complete` exist and stay anonymous only while setup is incomplete
- admin auth and route gating honor `setupRequired`
- setup wizard in the admin shell is capability-first and aligned with product defaults
- `/providers` remains in the product, but as advanced settings + provider docs rather than first-run onboarding
- focused tests cover setup state, auth state, provider config application, and controller gating

## Residuals Kept Explicit

- provider changes still require restart before runtime source/client wiring is guaranteed to switch fully
- `/providers` still exposes baseline/persisted/effective values because advanced operators need that visibility
- provider capability setup remains intentionally bounded; block-backfill policy and generic routing edits are still out of scope
- `Node ZMQ` remains an advanced post-setup path and is not exposed in wizard v1
- first-run setup does not attempt billing, account provisioning, or provider purchase automation

## Resulting Product Posture

- a new operator can configure `Consigliere` without understanding provider-specific jargon up front
- admin bootstrap is now aligned with the persisted operator-control model
- provider choice is explained by capability first, with advanced controls still available after setup
