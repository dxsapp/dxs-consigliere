# A1 Audit

## Result

Passed.

## What Was Checked

- history sync semantics remain honest:
  - `forward_only` never masquerades as full history
  - history reads require `acceptPartialHistory` for partial paths
  - `stateReadiness` can remain live while history is `backfilling_full_history` or `degraded`
- tracked history state is centralized in tracked docs/status docs instead of being scattered across controllers
- full-history execution state is separated into dedicated backfill job docs instead of overloading lifecycle timestamps alone
- historical address ingestion reuses canonical tx observation flow via `TxMessage.FoundInBlock`
- API surface stayed additive where possible:
  - registration requests gained nested `historyPolicy`
  - readiness/history responses gained explicit history status objects
  - upgrade operations are explicit endpoints rather than hidden re-registration semantics

## Open Watch Item

- `historical_token_scan` is now explicit in config/routing/contracts, but runtime execution is intentionally not implemented yet.
- current behavior is honest and deterministic:
  - token full-history jobs are queued
  - worker marks them `failed`
  - public `historyReadiness` becomes `degraded`
- this is acceptable for now because it is surfaced as an explicit unsupported path rather than a silent partial implementation.
