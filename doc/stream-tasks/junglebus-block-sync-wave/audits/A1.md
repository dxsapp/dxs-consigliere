# A1 Audit

## Decision

The wave is sufficient to treat JungleBus as the default block synchronization path for the product's first-run posture.

## Why

- block processing no longer begins with unconditional node RPC header lookup when `block_backfill` resolves to JungleBus
- the runtime now has a dedicated JungleBus block-sync monitor and scheduler that turn observed JungleBus block control messages into bounded `SyncRequest` ranges
- first-run setup explicitly collects JungleBus block-sync inputs and refuses completion when that capability is not configured
- startup/runtime messaging no longer implies that node RPC is required for the default bootstrap path

## Residual Advanced Node Role

Node infrastructure remains available for advanced operators:
- node RPC/ZMQ can still be configured outside the wizard
- node-backed chain-tip verification still runs when `block_backfill` primary resolves to `node`
- the wave does not remove node support; it removes node as a hidden prerequisite for the default path

## Residual Limits

- runtime/provider setup changes still require restart before all long-running background tasks pick up the new effective wiring
- the JungleBus block-sync monitor depends on a valid JungleBus `blockSubscriptionId`; the product now makes that dependency explicit instead of silently falling back to a broken node placeholder
- the repository still carries the existing `System.Text.Json 8.0.4` warning from `Dxs.Infrastructure`; this wave does not change that dependency
