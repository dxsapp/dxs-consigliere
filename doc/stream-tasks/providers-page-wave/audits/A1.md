# A1 Audit

## Verdict

`providers-page-wave` is acceptable to close.

The admin shell now has a dedicated `/providers` surface that:
- presents the supported provider landscape explicitly
- recommends `Bitails` for realtime and `WhatsOnChain` for REST
- allows bounded provider configuration without turning into a generic config editor
- separates provider onboarding/configuration from runtime diagnostics

## What Was Proved

- persisted provider override storage exists and remains bounded
- effective provider config is consumed by the live realtime and bounded REST-facing adapter settings paths
- `/api/admin/providers` exposes read/apply/reset contract for the page
- frontend admin shell renders a dedicated Providers page and keeps `/runtime` diagnostics-focused
- focused backend proof covers static vs override vs reset behavior and required-field rejection paths

## Residuals Kept Explicit

- runtime source changes still require restart before they are guaranteed to take effect fully
- historical address and token scans are still Bitails-backed and are not UI-switchable in this wave
- `/api/admin/runtime/sources` remains as a compatibility/runtime-summary surface and is no longer the main provider onboarding contract
- provider help links are onboarding aids only; there is no billing, purchase, or account provisioning flow inside the shell

## Resulting Product Posture

- new operators get one clear place to understand provider choices
- provider defaults now align with the intended product posture
- advanced sources remain available without becoming the default first-run experience
